# 5.2 — Design Effective Escalation and Ambiguity Resolution Patterns

## Concept

Agents must know when to stop and hand off to a human. Getting this wrong in either direction is costly: under-escalating causes harm, over-escalating destroys the value of automation. The key is building escalation on **explicit, deterministic rules** — not on vague heuristics or model confidence scores that fluctuate without explanation.

### When Agents Should Escalate

Three categories reliably warrant escalation:

**Confidence thresholds.** When an agent cannot resolve ambiguity after a bounded number of clarification attempts, it should stop rather than guess. The threshold should be defined per task type — a billing dispute tolerates far less uncertainty than a product FAQ response.

**Policy violations.** Certain topics or actions must always route to a human regardless of agent confidence: refund requests above a monetary threshold, account closures, legal complaints, anything touching protected personal data. These are rules, not judgments.

**Sensitive topics.** Crisis language, expressions of vulnerability, threats of self-harm, or regulatory subject matter (medical, legal, financial advice) require human judgment. The agent's job here is to detect and route, not to respond substantively.

### Designing Escalation Triggers

Escalation triggers must be **explicit and auditable**. A trigger like "escalate when the agent feels unsure" is untestable and will drift. A trigger like "escalate when intent confidence falls below 0.7 after two clarification turns" is measurable and improvable.

Define triggers as structured conditions evaluated at decision points in the agentic loop. Each trigger produces a typed escalation event with enough context for the receiving human to act immediately — no digging through logs required.

### Customer Preference Tracking

Not all customers want the same escalation behavior. Some want fast human handoff; others prefer the agent to exhaust all self-service options first. Store and apply these preferences explicitly. Preferences should be a first-class input to the escalation decision, not an afterthought.

A preference record might specify: preferred channel (chat, callback, email), escalation sensitivity (low / normal / high), and explicit opt-outs from specific automated flows.

### Why Sentiment Analysis Is Unreliable as an Escalation Signal

Sentiment scores are tempting because they feel intuitive — an angry customer should be escalated, right? In practice, sentiment analysis is a poor trigger for three reasons:

1. **Noisy signal.** Frustration in language does not reliably map to a customer who needs human intervention. Many frustrated customers are satisfied by a correct automated resolution.
2. **Latency mismatch.** Sentiment is evaluated on past utterances; the decision to escalate must anticipate the next moment in the conversation.
3. **Adversarial brittleness.** Customers who learn the system escalates on frustration will express frustration to bypass automation, degrading the signal further.

Use sentiment as a soft modifier at most — never as a primary trigger. Escalate on **what happened** (a policy threshold crossed, a failed resolution attempt, an explicit request), not on **how it sounded**.

### Structured Escalation Metadata

When an agent escalates, it must hand off context, not just control. A minimal escalation payload includes the reason (which trigger fired), the conversation summary, urgency level, and the customer's stated goal. This prevents the human agent from starting from scratch.

## Code Example

```typescript
// Escalation trigger definitions — explicit, not vibes
interface EscalationTrigger {
  readonly id: string;
  readonly reason: string;
  readonly urgency: "low" | "normal" | "high" | "critical";
}

const TRIGGERS = {
  LOW_CONFIDENCE: {
    id: "low_confidence",
    reason: "Intent confidence below threshold after clarification attempts",
    urgency: "normal",
  },
  POLICY_VIOLATION: {
    id: "policy_violation",
    reason: "Request exceeds automated handling policy",
    urgency: "high",
  },
  SENSITIVE_TOPIC: {
    id: "sensitive_topic",
    reason: "Topic requires human judgment",
    urgency: "critical",
  },
  CUSTOMER_PREFERENCE: {
    id: "customer_preference",
    reason: "Customer preference: always escalate this topic",
    urgency: "normal",
  },
} as const satisfies Record<string, EscalationTrigger>;

// Customer preference record
interface CustomerEscalationPreference {
  readonly customerId: string;
  readonly sensitivityLevel: "low" | "normal" | "high";
  readonly preferredChannel: "chat" | "callback" | "email";
  readonly alwaysEscalateTopics: readonly string[];
}

// Structured escalation event — carries full context for the human agent
interface EscalationEvent {
  readonly trigger: EscalationTrigger;
  readonly customerId: string;
  readonly conversationSummary: string;
  readonly customerGoal: string;
  readonly attemptsMade: number;
  readonly timestamp: string;
}

function buildEscalationEvent(
  trigger: EscalationTrigger,
  context: {
    customerId: string;
    conversationSummary: string;
    customerGoal: string;
    attemptsMade: number;
  }
): EscalationEvent {
  return {
    trigger,
    customerId: context.customerId,
    conversationSummary: context.conversationSummary,
    customerGoal: context.customerGoal,
    attemptsMade: context.attemptsMade,
    timestamp: new Date().toISOString(),
  };
}

// Escalation decision — deterministic, testable
function shouldEscalate(params: {
  intentConfidence: number;
  clarificationAttempts: number;
  topic: string;
  refundAmountUsd: number;
  preference: CustomerEscalationPreference;
}): EscalationTrigger | null {
  const { intentConfidence, clarificationAttempts, topic, refundAmountUsd, preference } = params;

  if (preference.alwaysEscalateTopics.includes(topic)) {
    return TRIGGERS.CUSTOMER_PREFERENCE;
  }

  const SENSITIVE_TOPICS = ["self-harm", "legal-complaint", "account-closure"];
  if (SENSITIVE_TOPICS.includes(topic)) {
    return TRIGGERS.SENSITIVE_TOPIC;
  }

  if (refundAmountUsd > 500) {
    return TRIGGERS.POLICY_VIOLATION;
  }

  const CONFIDENCE_THRESHOLD = 0.7;
  const MAX_CLARIFICATION_ATTEMPTS = 2;
  if (intentConfidence < CONFIDENCE_THRESHOLD && clarificationAttempts >= MAX_CLARIFICATION_ATTEMPTS) {
    return TRIGGERS.LOW_CONFIDENCE;
  }

  return null;
}
```

## Common Mistakes

**Using sentiment score as the primary escalation trigger.** Sentiment is noisy and gameable. Build triggers on structural facts: policy thresholds, topic classification, attempt counts.

**Escalating without context.** Handing off a customer ID and nothing else forces the human agent to re-read the full conversation. Always attach a summary, the customer's stated goal, and the reason for escalation.

**Treating escalation as failure.** Escalation is a designed outcome, not an exception. Agents that hide escalations or retry indefinitely to avoid them cause more harm than agents that escalate cleanly and early.

**Single global escalation threshold.** A 0.7 confidence threshold may be appropriate for order tracking but far too permissive for billing disputes. Define thresholds per domain.

**Ignoring customer preferences.** Some customers have indicated they want human contact for specific topics. Overriding that preference — even with a technically correct automated response — damages trust.
